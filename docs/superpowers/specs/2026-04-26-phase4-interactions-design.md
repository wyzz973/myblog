Incremental design for Phase 4 on top of the parent backend design and Phase 3's auth hardening. Adds reader-facing interactions (likes, comments) plus the admin-side moderation queue and activity stream. Synchronous SMTP lands in P4 (the ARQ-wrapped async version is still P5 work).

---

## 1. Goal

Move the wangyang.dev backend from "read-only public surface + admin CRUD" (P1+P3) to a fully interactive site:

- Real likes with IP+day idempotent dedup
- Public comment submission with admin moderation queue
- Public-endpoint rate limits (reusing the P3 Redis limiter)
- Synchronous SMTP for `magic_link` emails (upgrades P3 from log-only) and new comment notifications
- Activity-stream endpoints (powering admin dashboard widget + full activity log page)

After P4, every endpoint listed in the parent spec §4 (public) and §5 comments-backoffice / activity is implemented.

## 3. Data model changes

One Alembic revision: `0003_interactions`. Two new tables, no edits to existing tables.

### 3.1 `like_events`

```sql
CREATE TABLE like_events (
    id          INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    post_id     VARCHAR(64) NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
    ip_hash     VARCHAR(64) NOT NULL,                 -- sha256(ip + LIKE_SALT) hex
    day         DATE        NOT NULL,                  -- UTC date of the click
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT uq_like_events_post_ip_day UNIQUE (post_id, ip_hash, day)
);
CREATE INDEX ix_like_events_post_id ON like_events (post_id);
```

Like counts come from `SELECT count(*) FROM like_events WHERE post_id = $1`. The UNIQUE constraint enforces the per-IP-per-day idempotency at the DB level — `INSERT ... ON CONFLICT DO NOTHING` and the route returns whether the insert succeeded (`was_new`).

### 3.2 `comments`

```sql
CREATE TABLE comments (
    id          INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    post_id     VARCHAR(64) NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
    parent_id   INTEGER REFERENCES comments(id) ON DELETE CASCADE,   -- NULL = top-level
    who         VARCHAR(64) NOT NULL,
    email_hash  VARCHAR(64),                                          -- NULL for admin replies
    body        TEXT        NOT NULL,                                 -- plaintext, NOT markdown
    status      VARCHAR(16) NOT NULL DEFAULT 'pending',
    flag        BOOLEAN     NOT NULL DEFAULT FALSE,                   -- admin "flagged" tag
    actor       VARCHAR(8)  NOT NULL DEFAULT 'public',                -- 'public' | 'admin'
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT ck_comments_status CHECK (status IN ('pending','approved','spam')),
    CONSTRAINT ck_comments_actor  CHECK (actor IN ('public','admin'))
);
CREATE INDEX ix_comments_post_status ON comments (post_id, status);
CREATE INDEX ix_comments_status_created ON comments (status, created_at DESC);
CREATE INDEX ix_comments_parent_id ON comments (parent_id);
```

> **Schema notes:**
> - `parent_id` is a self-FK with `ON DELETE CASCADE` so deleting a parent removes its replies.
> - `email_hash = sha256(email.lower() + LIKE_SALT)` — raw email NEVER persisted.
> - `body` is plaintext only. The frontend renders it as text; no Markdown / HTML allowed.
> - `actor='admin'` rows are admin-authored replies created via `PATCH /admin/comments/{id} {reply_body: ...}`. Their `who` is the admin's display name (from `site_meta.name`), `email_hash` is NULL, `status` defaults to `'approved'`, `parent_id` references the comment being replied to.
> - Public submitters cannot specify `parent_id`; their comments are always top-level (`parent_id=NULL`).

## 4. Hashing strategy

Both like_events.ip_hash and comments.email_hash use the existing `LIKE_SALT` setting (P1 introduced it for likes; P4 extends usage). Construction:

```python
def _ip_hash(ip: str) -> str:
    return hashlib.sha256(f"{ip}|{settings.like_salt}".encode()).hexdigest()

def _email_hash(email: str) -> str:
    return hashlib.sha256(f"{email.lower().strip()}|{settings.like_salt}".encode()).hexdigest()
```

The pipe separator prevents trivial concatenation collisions (`a` + `b` vs `ab` + `_`). Raw IPs and raw emails never enter the DB or any log line.

## 5. Endpoints

All under `/api` (public) or `/api/admin` (auth-required) prefix.

### 5.1 Public — likes (`routers/public/posts.py` extended)

| Method | Path | Body / params | Response | Rate-limited |
|---|---|---|---|---|
| POST | `/posts/{id}/like` | — | `{likes: int, was_new: bool}` | yes — 10/min per (IP, post) |

Behaviour:
1. Resolve post by id (404 if not found, or `private=true`, or `status != 'published'`).
2. Compute `ip_hash` from `request.client.host`. Compute `day = today (UTC)`.
3. `INSERT INTO like_events (post_id, ip_hash, day) VALUES (...) ON CONFLICT DO NOTHING RETURNING id`.
4. Re-count `WHERE post_id = $1`.
5. Write `event_log` entry of type `post.liked` only when `was_new`.

### 5.2 Public — comments (`routers/public/comments.py`, new)

| Method | Path | Body | Response | Rate-limited |
|---|---|---|---|---|
| GET | `/posts/{id}/comments` | — | `[{id, who, body, created_at, admin_reply?: {id, who, body, created_at}}]` | no |
| POST | `/posts/{id}/comments` | `{who, email, body}` | `202 {id, status: "pending"}` | yes — 3/min per IP |

GET returns top-level approved comments (`status='approved' AND parent_id IS NULL`), each with up to 1 nested admin reply (`parent_id=this AND actor='admin' AND status='approved'`). At most one admin reply is shown per top-level comment.

POST always returns 202; on creation:
- Reject if `post.comments_enabled = false` → 403.
- Validate body fields (`1 ≤ who ≤ 64`, valid `email`, `1 ≤ body ≤ 4000`).
- Compute `email_hash`. INSERT row with `status='pending', actor='public', parent_id=NULL`.
- Fire `email.send_comment_notification(admin_email, comment_id, snippet)` — synchronous send (best-effort; on SMTP failure log warning, do not block the 202).
- Write `event_log` of type `comment.created`.

### 5.3 Admin — comment moderation (`routers/admin/comments.py`, new)

| Method | Path | Body | Response | Notes |
|---|---|---|---|---|
| GET | `/admin/comments?status=&post_id=&limit=&offset=` | — | `[{id, post_id, who, body, status, flag, parent_id, actor, email_hash, created_at}]` | filterable |
| PATCH | `/admin/comments/{id}` | `{status?, flag?, reply_body?}` | `{id, status, flag, reply_id?: int}` | reply creates a child comment row |
| DELETE | `/admin/comments/{id}` | — | 204 | hard delete + cascades replies |

PATCH semantics:
- If `status` given, validate transition (pending/approved/spam are all reachable from each other; admin override).
- If `flag` given, set boolean.
- If `reply_body` given, create a new comment row with: `parent_id=id, post_id=parent.post_id, who=site_meta.name, email_hash=NULL, body=reply_body, status='approved', actor='admin'`. Return `reply_id` of the new row.
- Multiple of the three may be combined in one call.

Both `current_admin` (JWT or write-scope api-token) is required; `Depends(require_scope("write"))` on PATCH/DELETE.

### 5.4 Admin — activity stream (`routers/admin/activity.py`, new)

| Method | Path | Params | Response |
|---|---|---|---|
| GET | `/admin/activity` | `type=` (optional repeat), `limit=50`, `offset=0` | `[{id, type, actor, target, meta, created_at}]` |
| GET | `/admin/dashboard/activity` | `limit=20` | `[{id, type, actor, meta, created_at}]` (no target/offset; for dashboard widget) |

Both query `event_log` ordered by `created_at DESC`. Read-only; no `require_scope` needed beyond `current_admin`.

## 6. SMTP integration

`app/services/email.py` is replaced (P3's log-only stub becomes a real client). The new shape:

```python
async def send_email(*, to: str, subject: str, body: str) -> None:
    """Send a plaintext email. Synchronous smtplib runs in asyncio.to_thread.

    Falls back to structlog.info() if smtp_host is None (dev mode), so the
    P3 magic-link tests still work without external SMTP.
    """

async def send_magic_link(*, email: str, url: str) -> None:
    subject = "Your wangyang.dev magic-link"
    body = f"Click to sign in: {url}\n\nValid for 15 minutes. If you didn't request this, ignore."
    await send_email(to=email, subject=subject, body=body)


async def send_comment_notification(*, to: str, comment_id: int, post_id: str, who: str, snippet: str) -> None:
    subject = f"[wangyang.dev] new comment on {post_id}"
    body = f"From: {who}\n\n{snippet[:280]}\n\nModerate: /admin#comments/{comment_id}"
    await send_email(to=to, subject=subject, body=body)
```

The `to_thread`-wrapped smtplib client uses STARTTLS by default; failures inside `send_email` are caught, logged at WARNING with `request_id`, and swallowed — comment submission/magic-link issuance never fails because of email transport.

### 6.1 Settings (additions to `app/config.py`)

```python
smtp_host: str | None = None                    # None = dev mode (log only)
smtp_port: int = 587
smtp_user: str | None = None
smtp_password: SecretStr | None = None
smtp_from: str = "noreply@wangyang.dev"
smtp_starttls: bool = True
admin_notify_email: str | None = None           # None → use accounts.email of the singleton admin
```

`admin_notify_email` lets the admin redirect comment notifications away from the login email if desired.

## 8. New event_log types

| Type | Actor | Target | Meta |
|---|---|---|---|
| `post.liked` | `ip_hash[:12]` | `post_id` | `{day}` |
| `comment.created` | `email_hash[:12]` | `comment_id` | `{post_id, who, length: int}` |
| `comment.moderated` | admin.email | `comment_id` | `{from_status, to_status}` |
| `comment.flagged` | admin.email | `comment_id` | `{flag: bool}` |
| `comment.replied` | admin.email | `comment_id` (parent) | `{child_id, post_id}` |
| `comment.deleted` | admin.email | `comment_id` | `{post_id}` |

These flow through to both `/admin/activity` and `/admin/dashboard/activity`. Combined with P3's auth events, the dashboard activity widget now has substantive content.

## 12. Acceptance criteria

P4 is complete when **all** of the following hold:

- [ ] Migration `0003_interactions` applies cleanly forward and backward
- [ ] All endpoints in §5 return the documented shapes; full pytest passes
- [ ] Like idempotency: same (IP, day) → second click returns same `likes` count
- [ ] Like rate-limit triggers 429 at 11th call within 60s
- [ ] Comment submitted as `pending` is INVISIBLE on public GET; appears on admin GET
- [ ] Admin PATCH `{status: 'approved'}` → public GET shows it
- [ ] Admin PATCH `{reply_body: '...'}` creates child row with `actor='admin', status='approved'`; public GET nests it under the parent
- [ ] DELETE cascades — orphan replies don't survive
- [ ] Comment 3/min rate limit triggers 429
- [ ] `comments_enabled = false` rejects POST with 403
- [ ] Email path: in dev (smtp_host=None) magic-link + comment notification log via structlog (no exception); with mock SMTP both call `smtplib.SMTP.send_message`
- [ ] No raw IP, raw email, or unhashed identifier appears in any DB column or any log line
- [ ] event_log gains rows for the 6 new types when corresponding events fire
- [ ] All P1/P2/P3 tests still pass; new tests cover the §9 list
- [ ] Spec §2 deferred items remain deferred (no ARQ, no automatic spam detection)
