Incremental design for Phase 5 on top of Phase 4. Adds the ARQ background worker, GitHub contribution sync, Anthropic-backed Pet LLM, and the "Now" history feature. Vercel deploy integration is **deliberately omitted** per the user's deployment strategy (self-hosted VPS, no Vercel).

---

## 1. Goal

Move the wangyang.dev backend from "interactive but isolated" (P4) to a fully integrated system:

- A standalone ARQ worker process running 7 scheduled / on-demand tasks
- GitHub PAT + GraphQL client → real `contrib_days` data feeding `/api/contrib`
- Anthropic API key + Claude calls powering `POST /api/pet/summon` (with admin-tunable fallback lines)
- `now_entries` table + admin/public endpoints for the "What I'm doing now" widget
- Migration of P3 magic-link email and P4 comment-notification email from synchronous `smtplib` to ARQ-enqueued background tasks (with retry + backoff)

After P5, every endpoint listed in the parent spec §4 (public Pet/Now/Contrib) and §5 admin integrations is implemented.

## 3. Data model changes

One Alembic revision: `0004_integrations`. Two new tables; no edits to existing tables.

### 3.1 `integrations`

```sql
CREATE TABLE integrations (
    name              VARCHAR(16) PRIMARY KEY,        -- 'github' | 'anthropic'
    username          VARCHAR(64),                    -- GitHub login; null for anthropic
    secret_encrypted  TEXT NOT NULL,                  -- AES-GCM via secret_box
    extra_json        JSONB NOT NULL DEFAULT '{}',    -- per-provider extras (e.g. anthropic.model)
    last_synced_at    TIMESTAMPTZ,
    last_status       VARCHAR(16),                    -- 'ok' | 'failed' | NULL
    last_error        TEXT,
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT ck_integrations_name CHECK (name IN ('github','anthropic'))
);
```

> **Deviation from parent spec §6.3**: parent spec §364 specifies Fernet for third-party API keys. P5 reuses **P3's `secret_box` (AES-GCM)** instead. Justification: secret_box is already an AEAD with semantic equivalence (12-byte nonce + GCM tag); introducing Fernet would mean two crypto primitives for the same job. The encrypted blob format is `base64(nonce || ciphertext || tag)` — same as P3 `tfa_secret_encrypted`.

### 3.2 `now_entries`

```sql
CREATE TABLE now_entries (
    id           INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    body_md      TEXT        NOT NULL,
    listening    VARCHAR(256),
    reading      VARCHAR(256),
    is_current   BOOLEAN     NOT NULL DEFAULT FALSE,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX ix_now_entries_one_current ON now_entries (is_current) WHERE is_current = TRUE;
```

> **Note**: the partial unique index enforces "at most one row with `is_current = TRUE`" at the DB layer. The admin endpoint un-flags any current row before flagging a new one within a transaction.

`listening` and `reading` are free-text strings (e.g. `"Boards of Canada — Geogaddi"`, `"《浮生六记》"`). No structured `{title, url}` shape — admin writes whatever they want.

### 3.3 `site_meta.pet_config` schema

Already a `jsonb` column from P1 (`default {}`). P5 fixes the schema admin endpoints assume:

```json
{
  "model": "claude-haiku-4-5-20251001",
  "system_prompt": "You are wangyang.dev's desktop pet ...",
  "fallback_lines": [
    "compiling thoughts...",
    "...",
    "..."
  ],
  "rate_limit_per_min": 6,
  "enabled": true,
  "species": "cat",
  "hat": "none",
  "tint": "#7aa7ff",
  "visitor_can_change": false
}
```

`PUT /api/admin/pet` validates and replaces the whole blob (Pydantic model). `GET /api/pet/config` returns only the public-safe subset: `{species, hat, tint, enabled, visitor_can_change}`.

## 4. ARQ workers framework

`arq` (Redis-backed) runs as a separate process: `arq app.workers.runner.WorkerSettings`. The web server (uvicorn) and worker do not share memory; they coordinate via Redis.

### 4.1 File layout

```
app/workers/
├── __init__.py
├── runner.py            (WorkerSettings: cron schedule + functions list)
├── queue.py             (enqueue helper used by routers/services)
└── tasks.py             (7 task implementations)
```

### 4.2 `runner.py`

```python
from arq.connections import RedisSettings
from arq.cron import cron
from app.config import get_settings

from app.workers.tasks import (
    publish_scheduled_posts,
    sync_github_contrib,
    cleanup_expired_magic_links,
    prune_event_log,
    send_email_task,
    recompute_post_word_counts,
)


class WorkerSettings:
    redis_settings = RedisSettings.from_dsn(get_settings().redis_url)
    functions = [
        send_email_task,
        recompute_post_word_counts,
        publish_scheduled_posts,
        sync_github_contrib,
        cleanup_expired_magic_links,
        prune_event_log,
    ]
    cron_jobs = [
        cron(publish_scheduled_posts, minute=set(range(0, 60))),       # every minute
        cron(sync_github_contrib, minute={5}),                          # :05 every hour
        cron(cleanup_expired_magic_links, minute={10, 40}),             # twice per hour
        cron(prune_event_log, hour={3}, minute={0}),                    # 03:00 UTC daily
    ]
    max_jobs = 4
```

### 4.3 `queue.py` — enqueue helper

```python
async def enqueue(name: str, *args, **kwargs) -> str:
    """Enqueue a task by registered name. Returns job_id.

    Routers/services call this instead of awaiting work directly.
    Falls back to inline execution when settings.arq_inline=True (test mode).
    """
```

In tests, `settings.arq_inline = True` makes `enqueue("send_email_task", ...)` execute synchronously in-process so HTTP integration tests don't need a worker. Production `arq_inline = False` does real Redis enqueue.

### 4.4 The 7 tasks (all in `tasks.py`)

| Task | Trigger | Behaviour |
|---|---|---|
| `publish_scheduled_posts` | cron 1/min | `UPDATE posts SET status='published' WHERE status='scheduled' AND scheduled_at <= now() RETURNING id` → write `post.published` event_log row per id |
| `sync_github_contrib` | cron :05 hourly + on-demand from PUT /admin/integrations/github | GraphQL query `contributionsCollection.contributionCalendar.weeks[].contributionDays[]` → upsert `contrib_days` (day, level, count). Update `integrations.last_synced_at` + `last_status` |
| `cleanup_expired_magic_links` | cron :10 :40 hourly | `DELETE FROM magic_links WHERE expires_at < now() OR consumed_at IS NOT NULL` |
| `prune_event_log` | cron 03:00 daily | `DELETE FROM event_log WHERE created_at < now() - interval '90 days'` (no archive table in P5; just hard delete; archive is P7) |
| `send_email_task` | on-demand (replaces P3 magic-link + P4 comment notification synchronous calls) | `await asyncio.to_thread(_send_sync, to=..., subject=..., body=...)` with ARQ's built-in retry-with-backoff (3 attempts) on SMTP error |
| `recompute_post_word_counts` | on-demand (called from `import-md` CLI) | recompute `word_count` for each post; update in batch |

### 4.5 Email migration (P3 + P4 backwards-compat)

`app/services/email.py` evolves:

```python
async def send_email(*, to: str, subject: str, body: str) -> None:
    settings = get_settings()
    if settings.smtp_host is None:
        log.info("email.dev_log", to=to, subject=subject,
                 body_sha256=hashlib.sha256(body.encode()).hexdigest()[:12],
                 body_len=len(body))
        return
    from app.workers.queue import enqueue
    await enqueue("send_email_task", to=to, subject=subject, body=body)
```

The ARQ task itself runs the existing `_send_sync` via `to_thread`. P3 + P4 callers (`send_magic_link`, `send_comment_notification`) stay unchanged because they call `send_email` — only the implementation behind it switches.

In test mode (`settings.arq_inline=True`), `enqueue` runs the task inline, so all P3 magic-link tests and P4 comment-notification tests continue to pass without modification.

## 5. GitHub integration

### 5.1 Endpoints (admin)

| Method | Path | Body / response | Notes |
|---|---|---|---|
| GET | `/api/admin/integrations/github` | `{username, last_synced_at, last_status, last_error}` | secret never returned |
| PUT | `/api/admin/integrations/github` | `{username, token}` → 200 with sync result, or 422 if GraphQL ping fails | upsert; ping then trigger sync inline |
| POST | `/api/admin/integrations/github/sync` | — → `{count, last_synced_at}` | manual re-sync; runs the same code path as the cron job, but synchronously (so admin sees the result) |

PUT validates by issuing one small GraphQL query (`viewer { login }`) with the new token. If 401, return 422 `{detail: "github token invalid"}` — don't store. On success, immediately invoke `sync_github_contrib(...)` inline (synchronous; ~1s for 52 weeks of data) so the admin sees fresh contrib_days right away.

### 5.2 GraphQL client

`app/services/github.py`:

```python
async def ping(token: str) -> str | None:
    """Return login on success, None on 401/403/network error."""

async def fetch_contributions(token: str, login: str, weeks: int = 52) -> list[dict]:
    """Returns [{day: date, count: int, level: int}] for the trailing N weeks."""
```

Uses `httpx.AsyncClient` against `https://api.github.com/graphql`. The level (0..4) is computed from `count` using GitHub's official thresholds (0, 1-3, 4-9, 10-19, 20+).

### 5.3 sync_github_contrib

```python
async def sync_github_contrib(ctx: dict) -> dict:
    """Upsert contrib_days from GitHub GraphQL. Returns {count, days_with_activity}."""
```

Reads the integration row from DB, decrypts token via `secret_box`, calls `fetch_contributions`, upserts `contrib_days` (PG `INSERT ... ON CONFLICT (day) DO UPDATE`). Updates `integrations.last_synced_at`, `last_status='ok'`, `last_error=None`. On exception, sets `last_status='failed'`, `last_error=str(e)[:512]`, re-raises so ARQ logs the failure.

### 5.4 `/api/contrib` cutover

Existing P1 endpoint serves seed data when `contrib_days` is empty. After P5, real GitHub data populates the table. The endpoint logic doesn't change — it just starts serving non-seed values.

## 6. Anthropic + Pet LLM

### 6.1 Endpoints

| Method | Path | Body / response | Notes |
|---|---|---|---|
| GET (admin) | `/api/admin/integrations/anthropic` | `{model, last_status, last_error}` | secret never returned |
| PUT (admin) | `/api/admin/integrations/anthropic` | `{api_key, model?}` → 200 / 422 | ping with a tiny `messages.create` call; reject if invalid |
| GET (admin) | `/api/admin/pet` | the full `pet_config` JSON | for editing |
| PUT (admin) | `/api/admin/pet` | full `pet_config` JSON → 200 | replaces entire blob |
| GET (public) | `/api/pet/config` | `{species, hat, tint, enabled, visitor_can_change}` | public-safe subset |
| POST (public) | `/api/pet/summon` | — → `{quip, source: 'llm' \| 'fallback'}` | rate-limited by IP |

### 6.2 Pet LLM call (`app/services/pet_llm.py`)

```python
async def summon(*, system_prompt: str, model: str, fallback_lines: list[str]) -> tuple[str, str]:
    """Call Anthropic; return (quip, source).

    On any error (no integration row, missing key, API failure, timeout),
    return a random fallback_line with source='fallback'.
    """
```

Uses the official `anthropic` SDK (already in spec deps line 492). Single-turn `messages.create` with:
- `system=system_prompt`
- `messages=[{role: "user", content: "summon"}]`
- `max_tokens=80`
- `temperature=0.9`
- 5-second timeout

### 6.3 Rate limit + event_log

`POST /api/pet/summon`:
- Rate-limit `rl:pet:{ip_hash[:16]}` at `pet_config.rate_limit_per_min` per minute (default 6).
- Write `pet.summoned` event_log row with `meta={source, ip_hash[:12]}`.
- Public response excludes the IP and includes only `{quip, source}`.

## 7. Now entries

### 7.1 Endpoints

| Method | Path | Body / response | Notes |
|---|---|---|---|
| GET (public) | `/api/now` | `{current: NowEntry, history: [NowEntry]}` (history limited to 10) | current = `is_current=TRUE` row; history = recent N approved-by-existence |
| GET (admin) | `/api/admin/now` | `[NowEntry]` (all rows desc) | full history for admin |
| POST (admin) | `/api/admin/now` | `{body_md, listening?, reading?, is_current=true?}` → `NowEntry` | if `is_current=true`, transactional flip of any prior current to false |
| PATCH (admin) | `/api/admin/now/{id}` | `{body_md?, listening?, reading?, is_current?}` → `NowEntry` | same flip logic |
| DELETE (admin) | `/api/admin/now/{id}` | 204 | hard delete |

### 7.2 NowEntry shape (response model)

```json
{"id": 1, "body_md": "Migrating the blog to Phase 5...", "listening": "Boards of Canada — Geogaddi", "reading": "《浮生六记》", "is_current": true, "created_at": "2026-04-26T..."}
```

### 7.3 Service module

`app/services/now.py` with `set_current(s, *, entry_id)`, `create(s, *, body_md, listening, reading, is_current)`, etc. All mutations write `now.created`/`now.updated`/`now.deleted` event_log rows in the same transaction as the data change (per P4 audit-atomicity pattern).

## 11. P3+P4 backwards compatibility

Critical: every existing test must keep passing.

- `app/services/email.py` `send_email` body changes (calls `enqueue` instead of `to_thread`). Tests use `arq_inline=True` so the actual SMTP call still runs in the same request flow.
- The P4 `test_email.py::test_smtp_mode_calls_smtplib` test patches `smtplib.SMTP`; with inline mode the patch still catches the call (the task runs in the same process).
- The P3 `test_auth_magic_link.py` tests assert on DB row creation, not log content — still pass.
- The P4 `test_post_comment_succeeds_when_smtp_down` test asserts 202 response on SMTP exception — still passes because the inline task still calls `_send_sync` which raises, then `send_email` swallows.

## 13. Acceptance criteria

- [ ] Migration `0004_integrations` applies cleanly forward and backward

- [ ] GitHub `PUT` with invalid token returns 422 and does NOT store the bad token

- [ ] All 8 new event_log types fire on the corresponding actions
- [ ] No raw third-party API key appears in any DB column or log line (only AES-GCM blobs)
- [ ] All P1-P4 tests still pass; new tests cover §10
